## What was done
- Modified `themes/neonbyte/components/header/primary-menu/primary-menu-wide.theme.css`: removed `[class]` attribute selector from `.primary-menu__link--level-2` and `.primary-menu__link--level-3` selectors, and removed `!important` from the `transition` declaration.

## Layer decisions
**CSS change in primary-menu-wide.theme.css (Layer 5 — component stylesheet):**

Trace:
1. **Config value?** No — this is a CSS property on a DOM element selector, not a Drupal config setting. → Not Layer 1.
2. **Theme token?** No — the transition values (`0.3s`) are hardcoded animation durations, not theme tokens. The `color` keyword references a CSS custom property but that's standard CSS, not a theme override. → Not Layer 3.
3. **Component-specific?** Yes — this rule targets `.primary-menu__link--level-2` and `.primary-menu__link--level-3`, which are component-level selectors within the primary-menu-wide component. The file `primary-menu-wide.theme.css` is a component theme stylesheet loaded via `libraries-extend`. → Layer 5 (correct).

**Why no override needed:** The base component CSS (`primary-menu-wide.css`) defines `.primary-menu__link` with specificity `(0,1,0)`. The theme CSS defines `.primary-menu__link--level-2` and `.primary-menu__link--level-3`, each also with specificity `(0,1,0)` (one class selector). Since both selectors have equal specificity and the `.theme.css` file loads **after** the base `.css` file in the library cascade, the theme rule naturally wins by source order. No `!important` or `[class]` attribute selector was needed.
