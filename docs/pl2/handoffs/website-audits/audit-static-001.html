CRITICAL themes/neonbyte/components/header/primary-menu/primary-menu-wide.theme.css:~line with transition property 
.primary-menu__link--level-2[class],
.primary-menu__link--level-3[class] {
  ...
  transition: --link-text-cover-percent 0.3s, color 0.3s !important;
}

An !important declaration on the transition property within a component's .theme.css file. There is no valid reason to use !important in this codebase — its presence means the developer is at the wrong layer or fighting specificity elsewhere. This forces every future override of the transition behavior on dropdown menu links to also use !important, creating a cascade debt spiral.

Remove the !important. If the transition is being overridden by a higher-specificity rule elsewhere, fix that rule's specificity or layer placement instead of escalating to !important.
INFO themes/neonbyte/css/themes/theme-white.css — selector :where(:root), .theme--white { } 
:where(:root),
.theme--white {
  /* Surfaces. */
  --theme-surface: var(--white);
  ...
}

The theme-white token definitions target both :where(:root) and .theme--white. The :where() function has zero specificity, which is technically correct for a token file. However, this pattern means the tokens are defined at :root unconditionally — they will be available even when no theme class is present on <html>. This can cause unexpected token values in contexts where the theme system hasn't applied its class yet (e.g., early render, admin overlay). The canonical Dripyard pattern scopes theme tokens under their respective theme selectors only.

Consider removing :where(:root) from the selector and relying solely on .theme--white for token definitions, ensuring tokens are only active when the theme system has explicitly applied its class.
INFO themes/neonbyte/css/_variables/variables-forms.css — selector :root, form { } 
:root,
form {
  --form-padding-inline: var(--sp1-5);
  ...
}

The form variables are defined on both :root and form. Since CSS custom properties inherit by default, defining them on :root alone is sufficient — all descendants including <form> will resolve them. The redundant form selector adds no functional value but increases specificity unnecessarily and may confuse future maintainers about whether form-specific scoping was intended.

Remove , form from the selector and keep only :root { }. If form-specific overrides are needed in the future, they should be scoped to form { } separately rather than duplicating root-level definitions.

Dimension: coupling-signals

WARNING themes/neonbyte/components/header/primary-menu/primary-menu-wide.css — .primary-menu__list--level-2 
.primary-menu__list--level-2 {
  width: calc(100% + var(--offset-from-header, 250px) - var(--header-padding-inline));
}

The dropdown menu width calculation compensates for a JS-set variable --offset-from-header. This creates a tight coupling between CSS layout and JavaScript state — if the JS fails to set this variable, the fallback of 250px is baked into the stylesheet, which may not match the actual offset. The comment acknowledges this: "The width goes to the edge of the header, so we take into account the --offset-from-header variable which is put in place by JS."

Consider whether the dropdown can be positioned using CSS containment or a container query instead of relying on a JS-injected offset. If JS coupling is unavoidable, document the expected range of values and add a CSS custom property fallback that matches the most common layout.
WARNING themes/neonbyte/components/header-article/header-article.css — .header-article__image img { margin-top: -150px; } 
.header-article__image[class] {
  ...
  img {
    width: 100%;
    margin-top: -150px;
  }
}

A hardcoded negative margin of -150px on the article image is compensating for spacing added at a parent level (likely the header-article's padding). This is a classic coupling signal — if the parent's spacing changes, this negative margin must also change. The value is not derived from any CSS variable or token.

Replace the hardcoded -150px with a CSS custom property (e.g., --header-article-image-offset) that can be adjusted alongside parent spacing changes, or use margin-block-start: calc(-1 * var(--spacing-xxxl)) if it corresponds to an existing token.
WARNING themes/neonbyte/components/hero/hero.css — .hero--position-against-screen-top 
&.hero--position-against-screen-top {
  padding-top: var(--space-for-fixed-header);

  :is(.dy-section, .layout-dynamic):has(&) {
    margin-top: calc(-1 * var(--space-for-fixed-header));
    overflow: revert;
  }
}

The hero component uses a negative margin on its parent section to compensate for the fixed header offset. While this is a documented pattern (the variable --space-for-fixed-header exists in variables-layout.css), it still represents coupling between the hero's positioning logic and the layout system's spacing tokens. If --space-for-fixed-header changes for any reason, both the padding and the negative margin must remain synchronized.

This pattern is acceptable if well-documented, but consider whether a CSS container query or inset-block-start on the hero itself could eliminate the need for parent-level negative margins. If keeping this approach, add a comment noting that both values must change together.
WARNING themes/neonbyte/css/layout/region.css — .region--fixed-middle-right { z-index: 20; } 
.region--fixed-middle-right {
  position: fixed;
  top: 50%;
  inset-inline-end: var(--drupal-displace-offset-right, 0);
  z-index: 20;
  transform: rotate(-90deg) translate(50%, 0%);
}

A z-index value of 20 on the fixed middle-right region (local actions tabs). This exceeds the recommended threshold of 10 and creates a stacking context that could conflict with other positioned elements. The comment explains it's for local actions tabs, which is a valid use case, but the high z-index means any future component using z-index: 15 or similar would be hidden behind this region.

If this region must sit above other content, consider whether a lower z-index (e.g., 10–12) combined with proper stacking context management via position and order would suffice. If 20 is necessary, add a comment documenting why this specific value was chosen and what other elements it must sit above.
WARNING themes/neonbyte/components/header/header-search/header-search.css — @media (forced-colors: active) 
@media (forced-colors: active) {
  background: none !important;
  aspect-ratio: revert;
  text-indent: revert;
}

An !important declaration inside a @media (forced-colors: active) block for Windows High Contrast Mode. While this is an accessibility override and the documentation exception applies, it still represents a cascade violation. The !important on background will prevent any future CSS from overriding the background removal in forced-colors contexts.

This is an acceptable accessibility override per the documented exception, but flag it for review: verify that revert on aspect-ratio and text-indent achieves the same goal without needing !important. If not, document why this specific !important is necessary for WCAG compliance.
