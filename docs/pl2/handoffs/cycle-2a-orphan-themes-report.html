Methodology — why no live-vs-baseline diff was run

The standard cycle-2b.1/2b.2 stash-baseline protocol could not be applied as-is: reproducing the pre-cycle rendering state requires re-importing 30+ config changes and re-creating two deleted on-disk theme directories. That is destructive and high-risk for a verification step.

The operator pre-authorized an alternative: declare AE = 0 by construction when no rendered HTML / CSS / Twig content was modified AND all 7 pages serve 200 with correct heading hierarchy. Both conditions hold:

  1. Zero .css files in the active theme performant_labs_20260502 were modified.
  2. Zero .twig templates in the active theme were modified.
  3. All four remaining config edits are either non-visual (metatag URLs in <head>, canvas-folder admin registry) or resolve to byte-identical referenced assets (favicon.svg recovered from 0ec999538, logo.svg already in active theme).
  4. T verified all 7 routes serve 200, all three referenced assets serve 200, and the active default theme is unchanged.

To corroborate, two consecutive live captures at 1280 were diffed against each other — AE = 0 confirms render stability and reliable baselining if needed in future.

Operator-acknowledged deviation: canvas.component.sdc.performant_labs_20260418.card-canvas.yml is retained with status: false instead of deleted, because deletion produces HTTP 500 on /services (Canvas page entity_id 3 has 4 component-tree entries referencing this SDC by ID). Cycle 2c will migrate those entries to sdc.dripyard_base.card-canvas and delete the orphan config then.
