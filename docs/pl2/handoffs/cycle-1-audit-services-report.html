What I see different — plain-English summary

  • Engagement cards (all viewports): live cards are skeletal — a thin top rule, transparent background, plain text. Preview cards have a soft cream surface fill with inner padding so each card reads as a contained block.
  • Engagement card numerals (all viewports): live reads "01 / TAKEOVER" in all-caps; preview reads "01 / Takeover" in title-case. Same casing difference on Embed / Pilot / a11y.
  • Engagement card titles (all viewports): preview h3 strings end with a period ("Test-suite takeover."); live versions have no period.
  • Logo strip (all viewports): live shows ~8 bitmap technology logos (Drupal, Playwright, Cypress, PHP, JS, React, Anthropic, plus one more) inside a "WE SPEAK"-labelled strip. Preview replaces this with a text-only inline wordmark row — six plain-text tool names spaced across a hairline-bounded strip, with "WE SPEAK" demoted INSIDE the strip as a small-caps label above the row.
  • Closing CTA (1280): live left-aligns the H2 with the primary button pushed to its right, and a separate secondary ghost CTA centered below. Preview centers the H2, places body copy under it, and aligns both buttons together in a single centered CTA row.
  • Closing CTA element order (1280): preview is kicker → H2 → body → buttons. Live is kicker → body → H2 → buttons (split layout).
  • Nearshore H2 wrap (1280): live wraps differently from preview because live uses the full page container width (~1140 px) whereas preview uses a content-cap around 640 px. Visible difference; everything else in this section matches.
  • Hero (all viewports): matches between live and preview at the brief spec (FU-2 canonical, excluded from rework).
  • Mobile (375): all sections collapse to single column on both. The card-surface and logo-strip deltas persist on mobile; the closing-cta layout delta collapses out.
Why the whole-page delta is 50–54 %: the preview is a standalone HTML file with minimal chrome; the live page wraps the same sections inside the full Drupal site header / breadcrumb / footer, which adds ~10–14 % more height and offsets every section below the hero by ~50–100 vertical pixels. After that offset, almost every pixel below the hero registers as a delta. The per-section delta classification below is the binding signal, not the whole-page percentage.

FU-6 answer — does the preview contain a heal-flow section?

No. The preview at docs/pl2/Previews/services.html contains five <section> elements:

  1. .hero — out of scope (FU-2 canonical).
  2. .engagements — four-card grid.
  3. .nearshore — cream capacity band.
  4. .proof — dogfooding pitch + "We Speak" text-only logo strip.
  5. .closing-cta — espresso final CTA.

No heal-flow, healing, or comparable section exists. The string healing appears only inside engagement-card #3 body copy ("Autonomous-healing pilot") and inside nearshore body copy.

Resolution per sprint runbook §"Approval Checkpoints": close FU-6 as "Services does not need heal-flow." Do not open a dedicated heal-flow cycle.
