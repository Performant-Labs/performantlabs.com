2 Warning

render-cascade — Variable Resolution

Checks: empty-string variables, wrong values, fallback-instead-of-primary

Warning
--theme-text resolves to empty string across entire page
--theme-text on canvas-page, theme--white, hero, card, heal-flow, title-cta (all viewports — all components)

The custom property --theme-text is captured as an empty string ("") on every sampled component at every viewport. This means no definition for this token reaches the computed style for any element on the page. If any component CSS references var(--theme-text) for text colour, it will silently resolve to its fallback or to initial with no browser error.

The theme-primary.css and theme-secondary.css files do not define --theme-text; the theme-zone variable files define --theme-text-color-soft, --theme-text-color-medium, and --theme-text-color-loud but not a bare --theme-text alias. Any future component that consumes --theme-text will receive nothing.

Computed value (all components, all viewports)
VariableExpectedComputedStatus
--theme-text a defined colour token "" (empty string) UNRESOLVED
Remediation
Define --theme-text in each theme-zone file (e.g. theme-primary.css) as an alias for the appropriate --theme-text-color-* token, or audit all component CSS files and replace any var(--theme-text) reference with the correct named token.
Warning
--color-brand-primary resolves to empty string across entire page
--color-brand-primary on all sampled components, all viewports

--color-brand-primary resolves to an empty string everywhere. This token is not listed in the known-context exclusions (only --theme-setting-* and --offset-from-header are Drupal-injected). The neonbyte and dripyard_base variable files do not define this property; it is neither derived from --primary nor aliased in any theme-zone file.

Any component consuming var(--color-brand-primary) for fill, border, or text will produce an invisible or initial-value result with no browser error. The broken-reference is latent — it will surface the moment a new component or style attempts to use it.

Computed value
VariableExpectedComputedStatus
--color-brand-primary a defined colour token "" (empty string) UNRESOLVED
Remediation
Define --color-brand-primary in themes/neonbyte/css/_variables/variables-colors-semantic.css as an alias for var(--primary), or remove all references to this undefined token and replace them with the canonical --primary or appropriate theme-zone alias.

wcag-contrast — Computed Colour Ratios

Ratios computed via WCAG relative luminance from captured rgb() values. Requirements: body text ≥ 4.5:1, large text (≥ 18pt / ≥ 14pt bold) ≥ 3.0:1, links ≥ 4.5:1.

Critical
Hero / title-cta CTA button: white text on teal background fails 4.5:1
.button — color: rgb(255,255,255) on background: rgb(98,187,203) — all viewports

The primary CTA buttons ("Book a testing review", "See how we test this site", "Or test with the tools") render white text over a teal surface. The computed contrast ratio is 2.19:1, far below the WCAG AA threshold of 4.5:1 for normal-weight body text at 15px.

This finding cross-references the axe-core flag noted in Known Context (hero primary button contrast). Numeric analysis confirms the failure independently.

Luminance calculation
ElementForegroundBackgroundL (fg)L (bg)RatioRequiredResult
.button rgb(255,255,255) rgb(98,187,203) 1.000 0.430 2.19:1 4.5:1 FAIL
Remediation
Darken the button surface token so white text achieves ≥ 4.5:1 (the teal would need to reach approximately oklch(0.48 0.08 200) or darker), or switch button text to a dark colour token that passes against the current teal. Update via the --theme-button-background-color token in the appropriate theme-zone CSS, not as a hardcoded hex.
Critical
Card links and icon-list links: teal text on white fails link contrast
.card__link, [icon-list inline link] — color: rgb(24,147,180) on background: rgb(255,255,255) — all viewports

All card "read more" links (.card__link) and the inline link within the icon-list section (the unnamed class [object key) render in teal rgb(24,147,180) against the white card or page surface. The computed contrast ratio is 3.55:1, below the WCAG AA 4.5:1 threshold required for link text at normal size.

The same teal is also used for .header-logo__link against the near-white header container (oklch(0.999994 …) ≈ white), producing the same 3.55:1 ratio. The logo link wraps an image so rendered text is absent there, but the computed colour is still out of range for any text fallback.

Darken the link colour token to at least oklch(0.46 0.10 213) (approximately rgb(0,110,140)) so it achieves 4.5:1 against white. Apply the change via the --theme-link-color token in the white and default theme-zone files, not with a hardcoded value in component CSS.
Warning
Footer column heading wrapper text is muted — verify it carries no meaningful label text
.footer-column__heading (wrapper) — color: rgb(92,84,76) on white — 12px font

.footer-column__heading renders at 12px in rgb(92,84,76) against white, giving a contrast of 7.38:1 which passes 4.5:1. However the font size is 12px (9pt), which is smaller than body text and the heading element itself appears to be a visual grouping label. The child .footer-column__heading-link element (also 12px, rgb(42,37,32)) passes at 15.9:1.

This is a warning rather than a critical because the wrapper node's colour passes numerically. The concern is that the 12px size places it just below the "large text" threshold where a 3.0:1 ratio would suffice — at this size the 4.5:1 threshold is the governing requirement, and it passes. Flag for visual QA to confirm the text carried in this element is decorative rather than informational.

Remediation
No immediate fix required. Confirm in visual QA (S) that .footer-column__heading nodes contain only non-informational decoration; if they carry visible label text, increase the type size to ≥ 14px to improve legibility at the current contrast level.

Critical
Hamburger menu still visible at 768px — breakpoint does not fire until 1280px
nav.hamburgerVisible: true at 768px — expected: false at ≥ 992px

The navbar-expand-lg pattern is documented to switch from hamburger to full inline navigation at 992px. At the 768px capture, hamburgerVisible is true — the hamburger remains visible. At 1280px it is false. This means the navigation is in hamburger mode all the way from the smallest mobile viewport through 768px (tablet landscape territory), and only collapses to inline at ≥ 1280px.

This is a critical responsive regression: tablet users at 768–991px see the hamburger icon and must tap to open a full-screen drawer rather than seeing the six inline nav labels. The design brief specifies the breakpoint at 992px; the rendered output does not match.

Remediation
Verify the navbar-expand-lg Bootstrap/Drupal breakpoint value in the theme's SCSS or CSS. The @media (min-width: 992px) rule that hides the hamburger and shows the nav list must be in effect; check for a conflicting override that raises the breakpoint beyond 992px and remove it.

The skip-to-main-content link has a rendered bounding box of 42×18 CSS px. The 18px height is critically below the 44×44 CSS px minimum. Its rendered tap target falls below WCAG 2.5.5 (AAA) and the practical 44px recommended minimum.

The width of 42px also narrowly misses the 44px minimum. Both dimensions need correction.

Out of Scope but Noticed

  • Render key [object in JSON: the Playwright capture script serialises a DOM object as a key name rather than the element's class string for one icon-list link entry. This means that element's class is not captured cleanly. A future audit run should verify the selector extraction logic in scripts/render-inspect.cjs to ensure element.className is coerced to a string before use as a JSON key.
  • heal-flow theme-surface vs rendered background mismatch: --theme-surface resolves to #F5EFE2 (cream) within .heal-flow's scope but the computed background of the element is white. A targeted static audit of the heal-flow component CSS would confirm whether an explicit background: #fff or background: var(--white) is overriding the token.
  • footer-column__link height inconsistency across viewports: the 44px height at 768px appears to be an accidental consequence of the column layout rather than an intentional design decision. A future audit should verify whether this is grid-driven and whether it collapses at other intermediate widths (e.g. 900px).
  • Unnamed bare <a> elements in footer: two interactive elements in the footer carry no BEM class (cls: ""). These are invisible to the Layer 5 CSS system; any future styling or touch-target fix requires a Twig template edit to add classes. A component-schema audit of the footer Twig would identify whether these are template gaps or Canvas-rendered links without slot class propagation.
  • --space-for-fixed-header: 160px resolved everywhere: this value matches what the JS-injected --offset-from-header pattern suggests, but it is a static fallback value (not JS-measured). If the header height changes (e.g. on scroll or orientation change), components relying on this token will not track the real header height. A future sprint should audit whether a JS-updated property should replace the static fallback.
